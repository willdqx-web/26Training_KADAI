# 要件定義書の書き方

---

## 1. 要件定義書とは何か

要件定義書とは、**「何を作るか」と「どう作るか（の骨格）」を文書化したもの**である。

実装を始める前に作成し、開発チーム全員が同じ認識を持つために使う。

---

## 2. 要求仕様書との違い

| 文書 | 目的 | 視点 | 例 |
|---|---|---|---|
| 要求仕様書 | 何を作るか | ユーザー・依頼者 | 「問い合わせを入力できること」 |
| 要件定義書 | どう作るかの骨格 | 開発者 | 「POSTリクエストでFastAPIに送信し、JSONに保存する」 |

要求仕様書で「何が必要か」を確認してから、要件定義書で「どう実現するか」を決める。

---

## 3. 要件定義書の構成

要件定義書には、以下のセクションを含めることが多い。

| セクション | 内容 |
|---|---|
| システム概要 | システム全体の構成・技術スタック |
| 機能要件 | 各機能が「何をするか」の具体的な定義 |
| 非機能要件 | 性能・信頼性・セキュリティなどの品質要件 |
| 画面一覧 | 作成する画面の一覧 |
| API一覧 | 作成するAPIのエンドポイント一覧 |
| データ仕様 | 保存するデータの構造・形式 |
| 対象外 | 今回は実装しないことを明示した項目 |

すべてのセクションが必須ではない。プロジェクトに応じて必要なものだけ書く。

---

## 4. 各セクションの書き方

### 4.1 システム概要

システムの全体像と技術スタックを書く。

**書くこと：**
- どのコンポーネントが何の役割を持つか
- コンポーネント間のデータの流れ（図で書くと分かりやすい）
- 使用する技術（言語・フレームワーク・ツール）

**書き方の例：**

```
[Streamlit] → HTTP → [FastAPI] → [JSONファイル]
                              ↓
                          [Gemini API]
```

```
| レイヤー     | 技術       |
| フロントエンド | Streamlit |
| バックエンド  | FastAPI   |
| データ保存   | JSONファイル |
```

---

### 4.2 機能要件

各機能が「何をするか」を具体的に書く。

**書くこと：**
- ユーザーが行う操作
- システムが行う処理
- 入力・出力の形式
- バリデーション（入力チェック）
- エラー時の動作

**良い書き方：**

```
### 問い合わせ入力機能
- テキストエリアで問い合わせ文を入力できる
- 送信ボタン押下でバックエンドにPOSTリクエストを送信する
- 入力が空の場合は送信できない
- 送信後は入力欄をクリアする
```

**悪い書き方（曖昧すぎる）：**

```
### 問い合わせ入力機能
- 問い合わせを入力できるようにする
- うまく動かす
```

**ポイント：** 「〜できる」「〜する」「〜しないこと」のように、動作が明確に分かる形で書く。

---

### 4.3 非機能要件

機能そのものではなく、品質・制約に関する要件を書く。

| 種類 | 何を書くか | 例 |
|---|---|---|
| 性能 | 処理速度・件数の目安 | 「1,000件以下は支障なく動作すること」 |
| 信頼性 | エラー時の動作 | 「APIエラー時はエラーメッセージを表示する」 |
| セキュリティ | 機密情報の扱い | 「APIキーは.envで管理しソースに書かない」 |
| 保守性 | コード構成の方針 | 「フロントとバックは別ファイルで管理する」 |

---

### 4.4 画面一覧

作成する画面を一覧にまとめる。

```
| 画面名          | 説明                    |
| 問い合わせ入力画面 | 問い合わせ文を入力し送信する |
| 一覧画面        | 登録済みの問い合わせを表示する |
| 詳細画面        | 特定の問い合わせの詳細を表示する |
```

---

### 4.5 API一覧

FastAPIで作成するAPIのエンドポイントを一覧にまとめる。

```
| メソッド | パス               | 機能                        |
| POST   | /inquiries         | 問い合わせを保存する           |
| GET    | /inquiries         | 問い合わせ一覧を返す           |
| GET    | /inquiries/{id}    | 指定IDの問い合わせ詳細を返す    |
```

**ポイント：**
- `GET` はデータを取得するときに使う
- `POST` はデータを新規作成・送信するときに使う
- パスの `{id}` はURLにIDを埋め込む場合の書き方

---

### 4.6 データ仕様

保存するデータの構造をJSONの例で書く。

```json
[
  {
    "id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "created_at": "2026-05-11T10:00:00+09:00",
    "content": "有給休暇の申請方法を教えてください。",
    "category": "休暇",
    "urgency": "低",
    "answer": "有給休暇の申請は..."
  }
]
```

各フィールドについて以下を記載する。

| フィールド名 | 型 | 内容 |
|---|---|---|
| id | 文字列（UUID） | 問い合わせID（自動採番） |
| created_at | 文字列（ISO 8601） | 登録日時 |
| content | 文字列 | 問い合わせ本文 |
| category | 文字列 | AIが判定したカテゴリ |
| urgency | 文字列 | AIが判定した緊急度（高／中／低） |
| answer | 文字列 | AIが生成した回答案 |

---

### 4.7 対象外の明示

「今回は実装しない」と明示することも重要である。

対象外を書く理由：
- 「あれも作ればよかった」「言ってくれれば作った」という認識のズレを防ぐ
- 開発範囲を絞り、期間内に完成させるための判断を明文化する

**書き方の例：**

```
## 対象外
- 認証・ログイン機能
- 管理者向けの対応・ステータス管理機能
- クラウドへのデプロイ
- データベース（RDB/NoSQL）の使用
```

---

## 5. 要件定義書を書くときの注意点

### NG：曖昧な表現を使う

| 曖昧な書き方（NG） | 明確な書き方（OK） |
|---|---|
| きれいに表示する | 登録日時の降順で一覧表示する |
| エラーを処理する | APIエラー時にフロント画面にエラーメッセージを表示する |
| 適切に保存する | UUIDと登録日時を含むJSONとして追記保存する |
| 速く動く | 1,000件以下の場合に支障なく動作すること |

---

### NG：実装していないことを書く

要件定義書は「実装する内容」を書くための文書である。  
ツールスキル学習でどこまで作れるかを理解した上で、実現できる範囲で書く。

---

### NG：決まっていないことを書かない

決まっていない項目は「未定」と書くか、省く。  
あとで実装と矛盾が生じる原因になる。

---

## 6. 要件定義書を作るステップ

```
Step 1: 要求仕様書を読み、実装する機能を確認する
        ↓
Step 2: システム構成図を書く（誰が何と通信するか）
        ↓
Step 3: 機能ごとに具体的な動作を箇条書きにする
        ↓
Step 4: 画面・API・データ構造を書く
        ↓
Step 5: 非機能要件・対象外を書く
        ↓
Step 6: チームで読み合わせ、認識のズレがないか確認する
```

---

## 7. チェックリスト

要件定義書を作成したら、以下を確認する。

- [ ] システム全体の構成が図や表で示されているか
- [ ] 各機能の動作が「〜する」「〜できる」で具体的に書かれているか
- [ ] 曖昧な言葉（「適切に」「きれいに」「うまく」）が使われていないか
- [ ] 画面・API・データの対応関係が整合しているか
- [ ] 実装しないことが明示されているか
- [ ] チームメンバー全員が読んで同じ理解になるか
