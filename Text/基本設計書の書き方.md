# 基本設計書の書き方

---

## 1. 基本設計書とは何か

基本設計書とは、**「システム全体をどのような構造で作るか」を決める文書**である。

要件定義書で「何を作るか」が決まった後、実装の前に「全体をどう組み立てるか」を整理する。

開発チーム全員が同じ設計イメージを持てるようにするために使う。

---

## 2. 他の設計文書との違い

| 文書 | 目的 | 決めること |
|---|---|---|
| 要件定義書 | 何を作るかの骨格 | 機能・非機能・API一覧・データ構造 |
| 基本設計書 | 全体の構造を決める | 構成図・画面レイアウト・処理フロー・フォルダ構成 |
| 詳細設計書 | 実装できる粒度まで詳細化する | API仕様・プロンプト・コードの処理手順 |

基本設計書は「どう組み立てるか（大きな絵）」、詳細設計書は「どうコードを書くか（細かい仕様）」と覚えるとよい。

---

## 3. 基本設計書の構成

以下のセクションを含める。すべてが必須ではなく、プロジェクトに応じて選ぶ。

| セクション | 内容 |
|---|---|
| システム構成 | コンポーネント間の関係・通信経路・起動方法 |
| フォルダ・ファイル構成 | ソースコードのディレクトリ構造 |
| 画面構成 | 作成する画面の一覧とレイアウト |
| 画面遷移 | 画面間の移動ルート |
| 処理フロー | 各機能の処理の流れ（誰が何をするか） |
| データ設計 | 保存するデータの構造と各フィールドの意味 |
| エラー処理方針 | エラーが起きたときの対処の方針 |
| 環境変数・設定値 | 外部に出してはいけない値や設定値の管理方法 |

---

## 4. 各セクションの書き方

### 4.1 システム構成

コンポーネント（部品）の関係と通信経路を図で表す。

**書くこと：**
- どのコンポーネントが存在するか
- コンポーネント間のデータの流れ（矢印で表す）
- 使用するポート番号やプロトコル

**書き方の例：**

```
[ブラウザ]
    ↓ HTTP (localhost:8501)
[Streamlit フロントエンド]
    ↓ HTTP REST API (localhost:8000)
[FastAPI バックエンド]
    ↓               ↓
[JSONファイル]   [Gemini API]
```

**ポイント：**
- 矢印の向きはデータが流れる方向（リクエストを送る側 → 受け取る側）
- ポート番号を書くと、開発時に「どこに接続するか」が明確になる

---

### 4.2 フォルダ・ファイル構成

ソースコードをどのようなフォルダ・ファイルに分けるかを書く。

**書くこと：**
- フォルダの名前と役割
- 主要なファイルの名前と役割

**書き方の例：**

```
プロジェクトルート/
├── frontend/
│   └── app.py          # Streamlit アプリ
├── backend/
│   ├── main.py         # FastAPI エントリポイント
│   └── services/
│       └── gemini_service.py  # Gemini API 呼び出し
├── data/
│   └── inquiries.json  # 問い合わせデータ
├── .env                # APIキー（Gitに含めない）
└── pyproject.toml
```

**ポイント：**
- 役割ごとにフォルダを分ける（フロントエンド / バックエンド / データ）
- ファイル名のコメントには「何のファイルか」を一言で書く

---

### 4.3 画面構成とレイアウト

作成する画面を一覧にし、各画面のレイアウトをASCIIで表現する。

**画面一覧の書き方の例：**

```
| 画面名          | 説明                    |
| 問い合わせ入力画面 | 問い合わせ文を入力し送信する |
| 一覧画面        | 登録済みの問い合わせを表示する |
| 詳細画面        | 特定の問い合わせの詳細を表示する |
```

**レイアウトの書き方の例：**

```
┌─────────────────────────────┐
│  AI問い合わせ管理アプリ        │
│  ─────────────────────────  │
│  [サイドバー]                │
│   ・問い合わせ入力             │
│   ・問い合わせ一覧             │
│                              │
│  ## 問い合わせ入力             │
│  ┌───────────────────────┐  │
│  │ （テキストエリア）         │  │
│  └───────────────────────┘  │
│  [送信する]                  │
└─────────────────────────────┘
```

**ポイント：**
- 実際の見た目をイメージして書く。完全な正確さは不要
- ボタン・テキストエリア・メニューなどのUIパーツが分かればよい

---

### 4.4 画面遷移

画面間の移動ルートを矢印と条件で表す。

**書き方の例：**

```
問い合わせ入力画面
        |
        | サイドバーで「問い合わせ一覧」を選択
        ↓
一覧画面
        |
        | 「詳細を見る」ボタンを押す
        ↓
詳細画面
        |
        | 「← 一覧に戻る」ボタンを押す
        ↓
一覧画面
```

**ポイント：**
- 矢印の横に「どの操作をすると遷移するか」を書く
- 「戻る」の遷移も忘れずに書く

---

### 4.5 処理フロー

各機能の処理の流れを書く。誰（どのコンポーネント）が何をするかを明確にする。

**書き方の例：**

```
[社員] 問い合わせ文を入力して「送信する」を押す
        ↓
[Streamlit] 入力が空でないかチェック
        ↓（空の場合はエラー表示して停止）
[Streamlit] POST /inquiries にリクエスト送信
        ↓
[FastAPI] リクエストを受け取る
        ↓
[FastAPI] Gemini API に問い合わせ文を送信
        ↓
[Gemini] カテゴリ・緊急度・回答案を返す
        ↓
[FastAPI] 結果を JSON ファイルに保存
        ↓
[FastAPI] 保存データを Streamlit に返す
        ↓
[Streamlit] 送信完了と AI 生成結果を表示
```

**ポイント：**
- `[コンポーネント名]` を行頭に書くと、どのコンポーネントが処理するかが一目で分かる
- 条件分岐（エラー時など）も書いておく

---

### 4.6 データ設計

保存するデータの構造と、各フィールドの意味・生成タイミングを書く。

**書くこと：**
- ファイル名・保存場所・フォーマット
- 具体的なJSONのサンプル
- 各フィールドの名前・型・内容・誰が生成するか

**書き方の例：**

```json
[
  {
    "id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "created_at": "2026-05-11T10:00:00+09:00",
    "content": "有給休暇の申請方法を教えてください。",
    "category": "休暇",
    "urgency": "低",
    "answer": "有給休暇の申請は..."
  }
]
```

```
| フィールド名 | 型           | 生成元          |
| id          | 文字列（UUID）| バックエンドで自動生成 |
| created_at  | 文字列（日時）| バックエンドで自動生成 |
| content     | 文字列       | ユーザー入力     |
| category    | 文字列       | Gemini API     |
| urgency     | 文字列       | Gemini API     |
| answer      | 文字列       | Gemini API     |
```

---

### 4.7 エラー処理方針

エラーが発生したときに、どのコンポーネントがどう対処するかを一覧にする。

**書き方の例：**

```
| エラーの種類           | 発生箇所    | 対処方法                       |
| 入力が空              | Streamlit  | エラーメッセージを表示し、送信しない |
| Gemini API 接続失敗   | FastAPI    | 503 を返す。Streamlit でエラー表示 |
| JSONファイルが存在しない | FastAPI    | 空のリストとして処理を続行する      |
```

**ポイント：**
- エラーの種類・発生場所・対処の3点を書く
- ユーザーに何を見せるかまで書くと実装しやすくなる

---

## 5. 基本設計書を書くときの注意点

### NG：コードを書く

基本設計書に実際のPythonコードを書くのは詳細設計書の役割である。  
基本設計書では「何をするか」を自然言語や図・表で書く。

### NG：処理フローが「〜する」だけで終わる

誰がどのコンポーネントに何を渡すかが分からないと実装できない。  
フローには必ず `[コンポーネント名]` を付けて書く。

### NG：エラー処理を書かない

「正常に動いたとき」だけ書いて終わりにしない。  
エラーが起きたときの動作も設計しておく。

---

## 6. 基本設計書を作るステップ

```
Step 1: 要件定義書の機能要件・API一覧・データ仕様を読み返す
        ↓
Step 2: システム構成図を書く（コンポーネント・通信・ポート）
        ↓
Step 3: フォルダ・ファイル構成を決める
        ↓
Step 4: 画面レイアウトと画面遷移を書く
        ↓
Step 5: 各機能の処理フローを書く
        ↓
Step 6: データ設計・エラー処理方針を書く
        ↓
Step 7: チームで読み合わせ、認識のズレがないか確認する
```
