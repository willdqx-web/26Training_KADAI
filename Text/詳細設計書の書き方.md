# 詳細設計書の書き方

---

## 1. 詳細設計書とは何か

詳細設計書とは、**「実際にどのコードを書くか」を決める文書**である。

基本設計書でシステムの全体構造が決まった後、各機能を「このように実装する」と具体的に定義する。

詳細設計書を読めば、実装担当者が迷わずコードを書ける状態を目指す。

---

## 2. 他の設計文書との違い

| 文書 | 粒度 | 具体例 |
|---|---|---|
| 基本設計書 | 全体の構造 | 「POST /inquiries でデータを受け取り、Geminiに送って保存する」 |
| 詳細設計書 | 実装できる粒度 | リクエスト・レスポンスのJSON形式、プロンプト全文、処理手順のステップ |

基本設計書は「何をするか」、詳細設計書は「具体的にどうするか」と覚えるとよい。

---

## 3. 詳細設計書の構成

以下のセクションを含める。すべてが必須ではなく、プロジェクトに応じて選ぶ。

| セクション | 内容 |
|---|---|
| API仕様 | 各エンドポイントのリクエスト・レスポンスの詳細 |
| データモデル | コードで使うデータ構造の定義 |
| 外部連携仕様 | Gemini APIなど外部サービスとの接続仕様 |
| データ操作処理 | JSONファイルの読み書き処理の手順 |
| 画面詳細 | 各画面のコンポーネントと処理の詳細 |
| エラー処理一覧 | エラーの種類・発生箇所・対処・ユーザー表示の対応表 |
| テスト観点 | 正常系・異常系・品質確認のテスト項目 |

---

## 4. 各セクションの書き方

### 4.1 API仕様

APIの各エンドポイントについて、リクエストとレスポンスの形式を具体的に書く。

**書くこと：**
- HTTPメソッドとURL
- リクエストのContent-TypeとJSONの形式
- 各フィールドの型・必須かどうか・説明
- 成功時のレスポンス（ステータスコードとJSONの形式）
- エラー時のレスポンス（ステータスコードと発生条件）

**書き方の例：**

```
### POST /inquiries　問い合わせ登録

リクエスト：
  POST http://localhost:8000/inquiries
  Content-Type: application/json

  {
    "content": "有給休暇の申請方法を教えてください。"
  }

  | フィールド | 型   | 必須 | 説明              |
  | content  | 文字列 | ○   | 問い合わせ本文（空不可）|

レスポンス（成功 201）：
  {
    "id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "created_at": "2026-05-11T10:00:00+09:00",
    "content": "有給休暇の申請方法を教えてください。",
    "category": "休暇",
    "urgency": "低",
    "answer": "有給休暇の申請は..."
  }

エラーレスポンス：
  | ステータス | 発生条件                    |
  | 422      | リクエストのJSON形式が不正     |
  | 503      | Gemini APIへの接続に失敗     |
  | 500      | GeminiのレスポンスがJSONでない |
```

**ポイント：**
- ステータスコードの意味を覚える

| コード | 意味 | 主な用途 |
|---|---|---|
| 200 | OK | GETの成功 |
| 201 | Created | POSTで新規作成の成功 |
| 404 | Not Found | 指定したリソースが存在しない |
| 422 | Unprocessable Entity | リクエストの形式が不正 |
| 500 | Internal Server Error | サーバー内部のエラー |
| 503 | Service Unavailable | 外部サービスへの接続失敗 |

### 4.2 データモデル

コードで扱うデータの型と構造を定義する。  
FastAPIではPydanticというライブラリを使ってデータモデルを定義する。

**書き方の例（Pydantic モデル）：**

```python
from pydantic import BaseModel

class InquiryRequest(BaseModel):
    content: str           # POST リクエストボディ

class InquiryRecord(BaseModel):
    id: str
    created_at: str
    content: str
    category: str
    urgency: str
    answer: str
```

**ポイント：**
- `InquiryRequest` はフロントエンドから受け取るデータ
- `InquiryRecord` はJSONファイルに保存・返却するデータ
- 詳細設計で型を明確にしておくと、実装時にミスが減る

### 4.3 外部連携仕様（Gemini API）

外部サービスとの接続仕様を書く。何を送り、何が返ってくるかを明確にする。

**書くこと：**
- 送信するプロンプトの全文
- Geminiに期待する出力形式
- 処理手順（ステップ形式）
- 想定されるエラーとその対処

**プロンプトの書き方の例：**

```
システムプロンプト：
  あなたは社内の総務部門向け問い合わせ分類AIアシスタントです。
  社員からの問い合わせ文を受け取り、以下のJSON形式のみで回答してください。
  余分な説明や前置きは不要です。

  出力形式：
  {
    "category": "カテゴリ名",
    "urgency": "高 または 中 または 低",
    "answer": "一次回答文"
  }

  カテゴリは以下の中から最も適切な1つを選んでください：
  勤怠, 休暇, 給与, 経費精算, 出張, 社員情報, その他

ユーザーメッセージ：
  {問い合わせ本文（そのまま渡す）}
```

**処理手順の書き方の例：**

```
1. 環境変数から GEMINI_API_KEY を取得する
2. google.generativeai.configure() で API キーを設定する
3. GenerativeModel("gemini-1.5-flash") でモデルを初期化する
4. システムプロンプトとユーザーメッセージを送信する
5. レスポンスのテキストを json.loads() でパースする
6. category / urgency / answer の3キーが存在するか確認する
7. 正常であれば dict を返す。失敗した場合は例外を raise する
```

**ポイント：**
- プロンプトには「JSON形式のみで回答」と明示する（余分な文章を防ぐ）
- プロンプトはコードに直接書かず、定数や別ファイルにまとめる

---

### 4.4 データ操作処理（JSON読み書き）

JSONファイルの読み込み・書き込みの処理手順を書く。

**全件読み込みの書き方の例：**

```
1. DATA_FILE_PATH のファイルが存在するか確認する
2. 存在しない場合は空のリスト [] を返す
3. 存在する場合は json.load() で読み込む
4. リストを返す
```

**1件追記保存の書き方の例：**

```
1. 全件読み込みを行いリストを取得する
2. 新しいレコードをリストの末尾に append する
3. json.dump() でファイルに上書き保存する
   （オプション：ensure_ascii=False, indent=2）
```

**ID検索の書き方の例：**

```
1. 全件読み込みを行いリストを取得する
2. id が一致するレコードを検索する
3. 見つかれば返す。見つからなければ None を返す
```

**ポイント：**
- 「存在しない場合」の処理を必ず書く（JSONファイルが初回は存在しない）
- `ensure_ascii=False` を付けないと日本語が文字化けする

### 4.5 画面詳細（Streamlit）

各画面のコンポーネントと処理の流れをコードに近い形で書く。  
完全なコードではなく「どのStreamlit部品を使い、何をするか」を書く。

**書き方の例（問い合わせ入力画面）：**

```
st.title("問い合わせ入力")

content = st.text_area("問い合わせ内容を入力してください", height=150)

if st.button("送信する"):
    if content.strip() == "":
        st.error("問い合わせ内容を入力してください。")
    else:
        response = POST /inquiries に content を送信
        if response.ok:
            data = response.json()
            st.success("送信が完了しました")
            st.write("カテゴリ：", data["category"])
        else:
            st.error("送信に失敗しました。")
```

**よく使うStreamlitの部品：**

| 部品 | 用途 |
|---|---|
| `st.title()` | 画面タイトル |
| `st.text_area()` | 複数行テキスト入力 |
| `st.button()` | ボタン |
| `st.success()` | 成功メッセージ（緑） |
| `st.error()` | エラーメッセージ（赤） |
| `st.info()` | 情報メッセージ（青） |
| `st.write()` | テキスト・データ表示 |
| `st.dataframe()` | テーブル表示 |
| `st.sidebar.radio()` | サイドバーのメニュー |
| `st.session_state` | 画面間で値を引き渡す |

### 4.6 エラー処理一覧

エラーの種類・発生箇所・対処・ユーザーへの表示を一覧にまとめる。

**書き方の例：**

```
| No | 発生箇所        | エラー内容              | 処理内容          | ユーザー表示                      |
| 1  | Streamlit      | 問い合わせ本文が空       | 送信せずに停止      | 「問い合わせ内容を入力してください。」  |
| 2  | FastAPI        | Gemini API 接続失敗    | 503 を返す        | 「送信に失敗しました。再試行してください。」|
| 3  | FastAPI        | GeminiがJSONを返さない  | 500 を返す        | 「送信に失敗しました。再試行してください。」|
| 4  | FastAPI        | JSONファイルが存在しない  | 空リストとして続行   | （エラーなし）                     |
| 5  | FastAPI        | 指定IDが存在しない      | 404 を返す        | 「問い合わせが見つかりません。」       |
```

**ポイント：**
- 「No」を付けると、テスト項目と対応づけやすくなる
- ユーザーへの表示文は具体的な文章で書く（「エラーが発生した」は曖昧）

---

### 4.7 テスト観点

実装後に確認するテスト項目をまとめる。

**正常系テスト（正しく動くかどうか）の書き方の例：**

```
| No | テスト内容                      | 確認ポイント                            |
| 1  | 問い合わせを送信する              | カテゴリ・緊急度・回答案が画面に表示されること |
| 2  | 送信後に一覧を表示する            | 送信した問い合わせが一覧の先頭に表示されること  |
| 3  | 一覧から詳細を表示する            | 全フィールドが正しく表示されること           |
```

**異常系テスト（エラー時の動作）の書き方の例：**

```
| No | テスト内容                      | 確認ポイント                            |
| 4  | 空のまま送信ボタンを押す           | エラーメッセージが表示され、送信されないこと  |
| 5  | JSONファイルがない状態で一覧を開く  | エラーにならず「履歴がありません」と表示されること |
```

**LLM回答品質確認の書き方の例：**

```
| No | テスト内容                           | 確認ポイント                           |
| 6  | 「有給休暇の申請方法を教えてください」を送信 | カテゴリが「休暇」と判定されること        |
| 7  | 「明日までに保険証の再発行が必要」を送信   | 緊急度が「高」と判定されること           |
| 8  | 回答案の内容を確認する                  | 問い合わせと無関係な回答でないこと        |
```

**ポイント：**
- 正常系・異常系・品質確認の3種類に分けて書く
- 「確認ポイント」は「〜されること」「〜でないこと」と具体的に書く

## 5. 詳細設計書を書くときの注意点

### NG：基本設計書と同じ粒度で止める

「Geminiに問い合わせを送る」は基本設計書で書くことである。  
詳細設計書では「どのモデルを使い・どのプロンプトを送り・どのキーを取り出すか」まで書く。

### NG：プロンプトを書かない

Geminiへのプロンプトは設計書に全文を書く。  
プロンプトが決まっていないと、実装者によって出力のフォーマットがバラバラになる。

### NG：エラー時の処理を「適切に対処する」で終わらせる

「適切に」では実装できない。  
どのHTTPステータスを返し、ユーザーに何を表示するかを具体的に書く。

## 6. 詳細設計書を作るステップ

```
Step 1: 基本設計書のAPI一覧・処理フロー・エラー処理方針を読み返す
        ↓
Step 2: 各APIのリクエスト・レスポンスの形式をJSONで書く
        ↓
Step 3: Pydanticモデルを書く
        ↓
Step 4: Geminiのプロンプト全文と処理手順を書く
        ↓
Step 5: JSON読み書きの処理手順を書く
        ↓
Step 6: 各画面のStreamlitコンポーネントと処理を書く
        ↓
Step 7: エラー処理一覧とテスト観点を書く
        ↓
Step 8: チームで読み合わせ、実装可能かどうか確認する
```
