# 共通準備資料

**uv・Sourcetree・WinMerge・ローカル環境の基本**

この資料は、AI問い合わせ管理アプリ開発研修の前に、全員が同じ開発環境とGit操作の基本をそろえるための資料です。

Git操作はSourcetreeを使用し、差分確認はWinMergeを使用します。uvのインストールはPowerShell（管理者）で行い、仮想環境はリポジトリをクローンした後に、そのリポジトリフォルダ内で作成します。

> 重要: この資料ではライブラリのインストールは行いません。uv pip installによるライブラリ追加は、後続のミニ課題で必要になったタイミングで実施します。

- [ ] PowerShell（管理者）でuvをインストールできる。

- [ ] エクスプローラーで作業用の親フォルダを作成できる。

- [ ] Sourcetreeで研修用リポジトリをクローンできる。

- [ ] クローンしたリポジトリフォルダへcdできる。

- [ ] uv venv -p 3.13で、リポジトリフォルダ内に仮想環境を作成できる。

- [ ] Sourcetreeで変更内容を確認できる。

- [ ] SourcetreeでAdd、Commit、Pushができる。

- [ ] SourcetreeでPullし、他メンバーの変更を取り込める。

- [ ] WinMergeを使って、変更前後の差分を確認できる。

- [ ] コンフリクトが発生したときに、内容を確認し、修正してCommitできる。

- [ ] .envや.venvなどGitに上げてはいけないファイルを理解できる。

> 注意: この資料では、Gitコマンドを覚えることよりも、Sourcetreeの画面で「何が起きているか」を理解することを重視します。

# 2. 今回使うツールの役割

| ツール | 役割 | 新人に理解してほしいこと |
| --- | --- | --- |
| uv | Pythonの仮想環境を作成する | プロジェクトごとにPython環境を分けるために使う |
| Sourcetree | Gitを画面操作する | Add、Commit、Push、Pullの流れを画面で理解する |
| WinMerge | ファイル差分を確認する | 自分が何を変更したか、衝突箇所はどこかを確認する |
| ターミナル | uv venvなどのコマンドを実行する | 作業場所を間違えないことが大切 |
| エクスプローラー | 作業用フォルダを作成・確認する | この資料ではmkdirコマンドでフォルダを作らない |

# 3. 最初に知っておくGit用語

| 用語 | 意味 |
| --- | --- |
| リポジトリ | ソースコードや変更履歴を管理する場所。研修ではai-inquiry-appがリポジトリになる。 |
| クローン | GitHubなどにあるリポジトリを自分のPCにコピーすること。 |
| Add | Commit対象にする変更を選ぶこと。 |
| Commit | 変更を履歴として保存すること。 |
| Push | 自分のCommitを共有先リポジトリに送ること。 |
| Pull | 他の人の変更を自分のPCに取り込むこと。 |
| Diff | 変更前と変更後の差分。 |
| コンフリクト | 複数人が同じ箇所を変更し、自動で統合できない状態。 |

# 4. uvをインストールする

uvは、Pythonの仮想環境作成やライブラリ管理を行うためのツールです。初回のみ、PowerShellを管理者として起動してインストールします。

1. Windowsのスタートメニューで「PowerShell」を検索する。

2. 「管理者として実行」を選択する。

3. 以下のコマンドを実行する。

```powershell
irm https://astral.sh/uv/install.ps1 | iex
```

インストール後、PowerShellを開き直して、以下でuvが使えることを確認します。

```powershell
uv --version
```

> 注意: uvのインストールはPowerShell（管理者）で実施します。ただし、日々の開発作業は通常のターミナルで構いません。

# 5. 作業用の親フォルダをエクスプローラーで作成する

作業用の親フォルダは、PowerShellのmkdirコマンドではなく、エクスプローラーで作成します。

1. エクスプローラーで作業場所を開く。例: C:\

2. 新しいフォルダーを作成し、フォルダー名を work にする。

3. 作成したフォルダーのパスを確認する。

```powershell
例: C:\work
```

> 重要: この段階では ai-inquiry-app フォルダを手動で作成しません。ai-inquiry-app フォルダは、次のSourcetreeのクローン操作で作成されます。

> 重要: この資料ではmkdirを使いません。backendやfrontendなどのフォルダを作る作業は、後続の別課題で実施します。

# 6. Sourcetreeでリポジトリをクローンする

先にリポジトリをクローンし、その後でリポジトリフォルダ内に仮想環境を作成します。この順番にすると、仮想環境がどのプロジェクト用なのか分かりやすくなります。

| 項目 | 指定例 |
| --- | --- |
| 親フォルダ | C:\work |
| クローン先 | C:\work\ai-inquiry-app |
| 仮想環境の作成先 | C:\work\ai-inquiry-app\.venv |

1. Sourcetreeを起動する。

2. 「Clone」を選択する。

3. 研修用リポジトリのURLを入力する。

4. 保存先に C:\work\ai-inquiry-app のように、リポジトリ名まで含めたパスを指定する。

5. Cloneを実行する。

6. クローン後、Sourcetree上でリポジトリが開けることを確認する。

> 避ける例: C:\work\ai-inquiry-app\ai-inquiry-app

> この形は ai-inquiry-app の中にさらに ai-inquiry-app ができてしまうため、研修では避けます。

# 7. クローンしたリポジトリ内で仮想環境を作成する

ターミナルでクローンしたリポジトリフォルダへ移動し、Python 3.13を指定して仮想環境を作成します。

```powershell
cd C:\work\ai-inquiry-app
uv venv -p 3.13
```

実行後、リポジトリフォルダ内に .venv フォルダが作成されていれば成功です。

```powershell
C:\work\ai-inquiry-app\.venv
```

> 補足: 作業フォルダの場所が異なる場合は、cdの後ろを自分のPC上のパスに置き換えてください。

> 重要: リポジトリをクローンする前に C:\work 直下へ .venv を作るのは避けます。仮想環境は、必ずクローンしたリポジトリフォルダの中に作成します。

# 8. この段階ではライブラリをインストールしない

この共通準備資料では、uv venv -p 3.13で仮想環境を作成するところまでを行います。

fastapi、uvicorn、streamlit、google-genai、openaiなどのライブラリは、後続のミニ課題で必要になったタイミングで個別に追加します。

> 理由: 最初にまとめてライブラリを入れると、「どの課題で何のために使うライブラリなのか」が分かりにくくなるためです。

この資料では、以下のようなコマンドは実行しません。

```powershell
# この段階では実行しない
uv pip install fastapi uvicorn streamlit requests google-genai python-dotenv pydantic pytest openai
```

# 10. Gitに上げないファイルを理解する

APIキーや仮想環境は、Gitに上げてはいけません。後続課題で.gitignoreを作成または確認するときは、以下が含まれているか確認します。

```
.venv/
.env
__pycache__/
data/*.json
data/*.csv
```

特に.envにはGemini APIキーなどを入れるため、誤ってPushしないように注意します。

# 11. WinMergeで差分を確認する

Diff確認はWinMergeを使用します。目的は「何を変えたのか」を自分で説明できるようにすることです。

1. Sourcetreeのオプション画面を開く。

2. 外部DiffツールにWinMergeを指定する。

3. 変更したファイルを選び、外部DiffでWinMergeを起動する。

4. 変更前と変更後の差分を確認する。

WinMergeの実行ファイル例:

```powershell
C:\Program Files\WinMerge\WinMergeU.exe
```

> 補足: メニュー名や設定画面はSourcetreeのバージョンにより異なる場合があります。研修では講師の画面を見ながら設定してください。

# 12. 基本操作: Add → Commit → Push

自分の変更をチームに共有する基本の流れです。

| 操作 | Sourcetreeで行うこと | 意味 |
| --- | --- | --- |
| Add | 変更ファイルを選び、ステージに追加する | Commitに含める変更を選ぶ |
| Commit | Commitメッセージを書いてCommitする | 変更を自分のPC上の履歴として保存する |
| Push | Pushボタンで共有先に送る | 自分のCommitをチームに共有する |

> 注意: Commit前には必ずWinMergeやSourcetreeの差分表示で、意図しない変更が含まれていないか確認します。

# 13. Commitメッセージの書き方

Commitメッセージは、あとで見返した人が変更内容を理解できるように書きます。

| 良い例 | 悪い例 |
| --- | --- |
| 問い合わせ入力画面のタイトルを変更 | 修正 |
| READMEに環境構築手順を追記 | 更新 |
| Gemini接続エラー時の表示文を追加 | いろいろ変更 |

> 目安: 「何を変更したか」が一目で分かる文にします。

# 14. Pullで他メンバーの変更を取り込む

チーム開発では、作業開始前やPush前にPullして、他メンバーの変更を取り込む習慣が重要です。

1. SourcetreeでPullボタンを押す。

2. 取り込み対象のブランチを確認する。

3. Pullを実行する。

4. 取り込み後にファイル内容が問題ないか確認する。

Pullしたあとに必要に応じてアプリやファイルを確認します。変更を取り込んだだけで安心せず、最低限の確認を行います。

# 15. コンフリクトが起きたときの対応

コンフリクトはミスではありません。複数人が同じ箇所を変更すると普通に発生します。大切なのは、慌てずに内容を確認して解消することです。

| 手順 | 内容 |
| --- | --- |
| 1 | Sourcetreeでコンフリクトが発生したファイルを確認する。 |
| 2 | WinMergeやエディタで、衝突している箇所を確認する。 |
| 3 | どちらの内容を残すか、または両方を統合するかをチームで決める。 |
| 4 | ファイルを正しい内容に修正する。 |
| 5 | コンフリクトマーカーが残っていないことを確認する。 |
| 6 | Sourcetreeで修正済みファイルをAddする。 |
| 7 | コンフリクト解消としてCommitする。 |

# 16. コンフリクトマーカーの例

コンフリクトが発生すると、ファイルに以下のような印が入ることがあります。

```
<<<<<<< HEAD
st.title("AI問い合わせ管理アプリ")
=======
st.title("総務問い合わせ管理アプリ")
>>>>>>> origin/main
```

この例では、2つのタイトル案が衝突しています。チームでどちらを採用するか決め、最終的に以下のように1つの内容に直します。

```
st.title("総務問い合わせ管理アプリ")
```

> 重要: コンフリクトマーカーが残ったままCommitしないでください。アプリが動かなくなる原因になります。

# 17. 演習: Git基本操作を体験する

以下の演習を行い、SourcetreeでAdd、Commit、Push、Pullを体験します。

| 演習 | 内容 | 確認すること |
| --- | --- | --- |
| 演習1 | READMEなどの練習用ファイルを1行変更する | Sourcetreeに変更が表示されるか |
| 演習2 | WinMergeで差分を確認する | どの行を変更したか説明できるか |
| 演習3 | 変更をAddしてCommitする | Commit履歴に残るか |
| 演習4 | Pushする | 共有先に反映されるか |
| 演習5 | 別メンバーの変更をPullする | 他メンバーの変更を取り込めるか |

# 18. 演習: コンフリクト対応を体験する

コンフリクト対応は、研修中に一度は意図的に体験します。

1. 2名が同じ練習用ファイルの同じ行を別々の内容に変更する。

2. 片方が先にCommit、Pushする。

3. もう片方がCommit後にPullする。

4. コンフリクトが発生することを確認する。

5. WinMergeで差分を確認する。

6. チームで採用する内容を決める。

7. ファイルを修正し、SourcetreeでAdd、Commitする。

> 重要: コンフリクト演習は練習用ファイルで行います。本番実装中の重要ファイルでは実施しません。
